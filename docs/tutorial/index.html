Product Overview

GHO Work is a desktop app for macOS and Windows that gives knowledge workers an AI agent capable of executing multi-step office tasks autonomously. It is powered by the GitHub Copilot SDK and connects to external services via MCP (Model Context Protocol) servers and CLI tools.

Core UX Principles

User in Control

Every action the agent takes requires explicit or pre-approved permission. The user can see, approve, deny, and audit everything.

Progressive Disclosure

Simple tasks are simple. Power features (custom skills, permission rules, hooks) reveal themselves as users grow.

Desktop-Native Feel

Keyboard-first, fast, responsive. Follows OS conventions for menus, shortcuts, and window management. Feels like VS Code, not a web app.

Transparent Agent Work

Tool calls, data access, and reasoning are always visible. No hidden actions. The tool activity panel shows everything.

Target Personas

DM

The Developer Who Also Manages

Software engineer or tech lead who already uses GitHub Copilot for code and wants the same AI-powered workflow for docs, reports, email, and project management.

Primary
PK

The Power Knowledge Worker

Tech-savvy professional working across multiple tools (Google Docs, Slack, Jira, spreadsheets) who values autonomy over IT-managed solutions.

Secondary
ST

The Startup Team

Small team (5-50) using mixed SaaS tools. Has GitHub Copilot for devs but can't justify M365 Copilot pricing for AI-assisted productivity.

Tertiary

Key Concepts

Concept What It Is Where in the UI
Workspace A folder on disk that scopes conversations, permissions, and memory files (CLAUDE.md) Workspace picker, sidebar header, status bar
Conversation A chat session with the agent. Persists across restarts. Contains messages and tool call history. Sidebar conversation list, chat panel
Connector An MCP server or CLI tool that gives the agent access to an external service Settings > Connectors, status bar indicators
Skill A reusable prompt template invoked via slash command (e.g., /draft-email) Chat input autocomplete, Skills panel
Tool A specific action the agent can take (read file, send Slack message, run shell command) Tool call cards in chat, Tool Activity panel
Permission Rule A saved decision about whether a tool is always allowed or always denied Permission dialogs, Settings > Permissions

Design Notes

  • The onboarding flow uses a single centered card layout — no sidebar, no panels.
  • Each step has a clear primary action button. Back navigation via browser-style back or breadcrumb.
  • The connector step is skippable — the CTA says "Start Using GHO Work" regardless of whether connectors were added.
  • On subsequent launches, the app opens directly to the workbench with the last workspace.
  • If the Copilot subscription check fails, show a clear error with a link to subscribe.

The chat panel is the primary interaction surface. It displays a threaded conversation with the AI agent, including streaming text, tool call cards, permission prompts, and rich content rendering.

Permissions System

Every tool call goes through the permission system. The user controls what the agent can do, with four decision levels: Allow Once, Allow Always, Deny, and Deny Always. This is the core trust mechanism.

Inline Permission Prompt

Appears directly in the chat flow when the agent wants to execute a tool that has no saved rule. The agent pauses until the user decides.
GHO Work — Permission Request
I'll send the report to the #sales channel now.
Permission Required
slack / postMessage via Slack MCP Server
channel: #sales
message: Weekly Sales Update (Mar 3-7) Great week, team! Here are the highlights: - Total revenue: $142,500 (+12% vs prior week) - New deals closed: 8 (up from 5) - Top performer: Sarah Chen — $48,200 - Pipeline growth: $320K added Keep the momentum going!
"Allow Always" will create a rule: slack/postMessage → always allow

Permission Decisions

Allow Once

Approve this specific tool call. No rule is saved. Next time the same tool is called, the prompt appears again.

Enter

Allow Always

Create a persistent rule. This tool (from this server) will be auto-approved in future. Rule is visible in Settings > Permissions.

Shift+Enter

Deny

Block this specific call. The agent is notified and will try an alternative approach or explain what it couldn't do.

Esc

Deny Always

Create a persistent deny rule. This tool will never be executed. The agent won't attempt it again.

Shift+Esc

Bash/Shell Permission

Custom Skill File Format

Custom skills are Markdown files with YAML frontmatter, stored in .claude/skills/ or .github/skills/.
.claude/skills/weekly-standup.md 
---
name: Weekly Standup Summary
description: Generate a standup summary from this week's activity
allowed-tools:
  - google-calendar/*
  - slack/searchMessages
  - github-issues/*
user-invocable: true
---

# Weekly Standup Summary

Generate a concise standup update based on this week's activity.

## Context

The user's calendar events this week:
```sh
# This command runs automatically to provide context
mgc users calendar events list --start "$(date -v-7d +%Y-%m-%d)" --end "$(date +%Y-%m-%d)" --select "subject,start,end" --output json
```

## Instructions

1. Check the calendar for this week's meetings
2. Search Slack for messages from the user this week
3. Check GitHub for issues/PRs the user worked on
4. Compile into a standup format:
   - **Done this week**
   - **In progress**
   - **Blockers**
   - **Next week priorities**

Task Queue

Unlike single-threaded chat, GHO Work supports a task queue. Users can submit new tasks while the agent is busy with a previous one. Tasks execute sequentially (or in parallel via subagents).

Task Queue UI

GHO Work — Task Queue

Task Queue

3 tasks
Analyze Q1 sales spreadsheet Running — 3 tool calls, 45s elapsed
2
Draft weekly update email Queued — waiting
✓
Summarize meeting notes from yesterday Completed — 2m 14s, 7 tool calls

Task Queue Behavior

  • New messages sent while the agent is working are queued as tasks.
  • The queue is visible in the status bar ("2 tasks queued") and expandable as a panel.
  • Tasks execute in order. Completed tasks link back to their conversation messages.
  • Users can cancel queued or active tasks without affecting other tasks.
  • Subagents can run in parallel for independent parts of a single task.

Hooks Configuration

Hooks allow custom scripts to run at defined points in the agent lifecycle. Configured in .claude/settings.json.

Available Hook Events

  • pre_tool_call — Runs before a tool is executed (can modify arguments or abort)
  • post_tool_call — Runs after a tool completes (can inspect results, trigger follow-up)
  • session_start — Runs when a new agent session begins
  • session_end — Runs when a session ends (cleanup, reporting)
.claude/settings.json
{ "hooks": { "pre_tool_call": [ { "matcher": "Bash", "command": "scripts/validate-command.sh", "timeout": 5000 } ], "post_tool_call": [ { "matcher": "*", "command": "scripts/log-tool-call.sh", "timeout": 3000 } ], "session_start": [ { "command": "scripts/setup-workspace.sh" } ], "session_end": [ { "command": "scripts/cleanup.sh" } ] } }

Hook Execution in Tool Activity

Hooks appear in the tool activity feed as distinct entries before and after the tool call they wrap.
pre_tool_call validate-command.sh passed
Bash npm run build success
post_tool_call log-tool-call.sh logged

Empty & Error States

Every view in GHO Work has a designed empty state that guides users toward their first action. Error states provide clear feedback with recovery options.

Empty States

Shown when a view has no content yet. Each empty state includes an icon, a short message, and a call-to-action.
Empty State — Conversations
No conversations yet
Start a new conversation with Cmd+N or type in the chat input
Empty State — Connectors
No connectors configured
Add your first MCP server to get started
Empty State — Documents
No files in workspace
Open a workspace folder to see your files here

Error States

Error states appear inline in the relevant context (chat area for agent errors, sidebar for connector errors). Each includes a clear description and a retry action.
