SubagentProgressBridge — Event Flow

How SDK events drive the info panel progress section

Event Mapping
SDK SESSION EVENTS
session.plan_changed
type: "created" | "updated" | "deleted"
content: "- [ ] Pull metrics\n- [ ] Draft slides..."
subagent.started
parentToolCallId, name, displayName, description
subagent.completed
parentToolCallId, name, displayName
subagent.failed
parentToolCallId, name, error
→
BRIDGE LOGIC
parsePlanMarkdown()
"- [ ] Pull metrics" → { id: "step-0", label: "Pull metrics", state: "pending" }
"- [x] Draft slides" → { id: "step-1", label: "Draft slides", state: "completed" }
correlateSubagentToStep()
Sequential: next pending step = active step
Attaches subagent displayName as badge
updateStepFromSubagent()
started → state: "active", badge: displayName
completed → state: "completed"
failed → state: "failed", error: message
→
INFO PANEL
ProgressSection
✓ Pull quarterly metrics
● Draft slide deck Doc Drafter
  Email team with summary
1 of 3 steps completed

Edge Cases

No plan created: Simple task — progress section stays hidden. No change from today.
Plan but no subagents: Main agent works through steps itself. Plan events still update the stepper. Steps transition via plan_changed (re-parse markdown for checkbox changes).
Subagent without plan: Model delegated without planning first. Show subagent activity as a single-step progress indicator ("Running: Doc Drafter"), not a stepper.
Plan changes mid-execution: Agent adds/removes steps. Re-parse markdown, preserve state of existing steps, add new ones as pending.
Subagent fails: Mark step as failed with error. Main agent decides whether to retry, skip, or abort — we don't intervene.
